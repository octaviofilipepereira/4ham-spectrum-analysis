1 · Overview

4ham Spectrum Analysis is a real-time spectrum monitoring and digital-mode decoding system for amateur radio, built around an SDR (Software Defined Radio) receiver. It continuously scans configured HF bands, decodes digital signals (FT8, FT4, WSPR, CW) and SSB occupancy, logs every detected event, and displays all activity on a live waterfall and propagation map.

The system has two parts:

  • Backend — Python/FastAPI server that drives the SDR, runs the decoders and stores events in SQLite.
  • Frontend — Single-page web application served by the backend; accessible on the local network at http://<host>:8000.

3 · Scan modes

Only one mode runs at a time on one band. Select the mode before (or during) a scan using the mode buttons.

ModeWindowMarker TTLNotes
FT8 15 s45 s Most popular weak-signal HF mode. Decoded with jt9. Shows callsign markers on the waterfall dial frequency.
FT4 7.5 s23 s Faster variant of FT8, also decoded with jt9. Higher throughput, slightly less sensitivity.
WSPR 120 s360 s Very slow beaconing mode, typically below the noise floor. 2-minute decode windows.
CW dwell 30 s45 s Morse code decoder. Steps across the band in configurable frequency steps. CW Step (Hz) and CW Dwell (s) parameters appear when CW is selected.

ParameterDefaultDescription

4 · Scan Rotation

Scan Rotation lets you define a sequence of slots (band + mode) that the system cycles through automatically, switching band and mode at configurable intervals. It is ideal for unattended monitoring of multiple bands over long periods.

Configuring rotation

  1. Click Config Scan Rotation in the scan toolbar.
  2. Select the rotation mode: Band + Mode (each slot defines band and mode) or Band only (rotates through bands, keeping the current mode).
  3. Choose a Band and Mode, then click + Add New Slot.
  4. Repeat to build your rotation sequence. Slots appear as editable badges — click × to remove one.
  5. Set the Dwell time (30 s, 1 min, 2 min, 5 min, 10 min, 15 min, or 30 min).
  6. Enable Loop to restart the sequence after the last slot, or disable it to stop at the end.
  7. Click Start Rotation.

Live status

While rotation is running:

  • A ● red pulsing dot and the current band + mode are shown in the rotation status bar.
  • A countdown timer shows the remaining time on the current slot.
  • The next slot (band + mode) is displayed for reference.
  • Click Stop scanning to stop the rotation and the current scan.

Data from all scanned slots accumulates in the database and appears in the propagation map and analytics dashboards.

💡 Tip: Combine rotation with the Academic Analytics dashboard to see propagation patterns across bands over time — the Heatmap Pro and activity timeline charts are particularly useful with multi-band rotation data.

4b · Rotation Presets & Scheduler

Rotation Presets let you save named rotation configurations (slot list, dwell time, loop mode) so you can quickly switch between different monitoring strategies. The Preset Scheduler takes this further by automatically activating presets based on the time of day (UTC).

Managing Presets

  1. Click Rotation Presets in the scan toolbar to open the Presets modal.
  2. The Available Presets section lists all saved presets with their slots summary.
  3. Click Load to apply a preset to the current rotation configuration.
  4. Click Delete to remove a preset you no longer need.
  5. To create a new preset, configure your rotation slots and dwell time in the Scan Rotation panel, then enter a name and click Save current config as preset.

Preset Scheduler

The Scheduler allows you to assign presets to specific time windows (UTC) so the system automatically switches rotation presets throughout the day — ideal for adapting to propagation changes between daytime and nighttime bands.

  1. In the Presets modal, scroll to the Preset Scheduler section.
  2. Select a preset from the dropdown, enter a Start and End time (HH:MM, UTC), then click Add Schedule.
  3. The schedule table shows all configured time windows, sorted by start time.
  4. Use the On/Off toggle to enable or disable individual schedules without deleting them.
  5. Click Start Scheduler to activate automatic preset switching.

The scheduler checks every 30 seconds which time window is active and applies the corresponding preset. If rotation stops unexpectedly, the scheduler detects and re-applies it automatically.

Schedule rules

  • Cross-midnight windows are supported (e.g. 22:00 → 06:00).
  • Overlapping windows are rejected — the system returns an error if a new schedule conflicts with an existing one.
  • The scheduler auto-starts on boot if there are enabled schedules in the database.
  • The current UTC time is displayed above the schedule table for reference.
💡 Tip: Create presets for daytime (e.g. 10m, 15m, 20m) and nighttime (e.g. 40m, 80m, 160m) bands, then schedule them to switch automatically — the system runs fully unattended 24/7.

7 · SSB Voice Signature & Whisper ASR

Since v0.8.0, the system can detect and transcribe SSB voice transmissions using VAD (Voice Activity Detection) and OpenAI Whisper automatic speech recognition.

How it works

  1. During scanning, SSB occupancy is detected by bandwidth and power heuristics.
  2. If voice activity is confirmed (VAD), a 15-second audio segment is captured.
  3. If Whisper ASR is enabled, the audio is transcribed in a background thread.
  4. The transcript is parsed for callsign patterns (NATO phonetics, direct callsigns, "CQ" calls).
  5. If a callsign is resolved it appears with the callsign; if a transcript exists but no callsign, it appears as Voice Transcript (click TXT to see the text); otherwise it appears as Voice Confirmed.

SSB event labels

SSB events are shown with one of three labels depending on the detection result:

LabelMeaning
Voice ConfirmedVoice activity detected by VAD but no Whisper transcript available.
Voice TranscriptWhisper produced a transcript (shown via the TXT button) but no valid callsign was resolved.
CallsignWhisper transcript contained a valid callsign — shown directly in the callsign column.

All three types are included in the "Show All" filter by default.

VOICE DETECTED waterfall markers

Confirmed SSB voice activity (ASR-verified) produces real-time VOICE DETECTED markers on the waterfall overlay. These markers have a distinctive black background with gold text and a 45-second TTL — long enough to survive natural pauses between SSB "overs" without flickering. Each new voice detection resets the timer. Since v0.8.4, markers are only created when Whisper ASR confirms actual voice; occupancy-only detections no longer produce waterfall markers.

Enabling Whisper ASR

Open Admin Config and scroll to SSB Voice Transcription (ASR). Tick the checkbox and click Save ASR setting. Whisper requires the openai-whisper Python package (~700 MB download). You can install it via the interactive installer or manually: pip install openai-whisper.

💡 On Raspberry Pi or low-end hardware, each 15 s audio segment takes 2–5 s to transcribe. Transcription runs in a background thread and does not block scanning.

8 · Metrics explained

SNR (Signal-to-Noise Ratio)

Instantaneous measurement of a single signal relative to the noise floor, in dB.

SNR rangeInterpretation
< 8 dBSignal rejected — too weak to decode reliably.
8–15 dBWeak but decodable.
15–25 dBStrong signal.
> 25 dBExcellent signal strength.

The SNR shown next to each band in the propagation map is the maximum peak SNR recorded in the last 60 minutes — not a current reading.

Propagation Score (0–100)

An aggregated quality score for a band, computed from all events in the last 60 minutes. Since v0.9.0 the system uses three distinct formulas tailored to each mode category:

CategoryModesPrimary Metrics
DigitalFT8, FT4, WSPR, JT65, JT9, FST4, FST4W, Q65Decode rate (40%), median SNR (35%), unique callsigns (15%), recency (10%)
CWCW, CW_CANDIDATE, CW_TRAFFICTraffic volume (30%), SNR (30%), signal strength (15%), callsign bonus (15%), recency (10%)
SSBSSB, SSB_TRAFFIC, AM, VOICE_DETECTIONTraffic (20%), SNR (25%), signal (15%), voice quality (20%), transcript (10%), callsign bonus (5%), recency (5%)

Each mode uses mode-specific SNR normalisation based on the actual decode threshold for that mode (e.g. FT8: −20 dB floor, CW: −15 dB, SSB: +3 dB).

ScoreClassification
≥ 70🟢 Excellent
50–69🟡 Good
30–49🟠 Fair
< 30🔴 Poor
⚠ A band can show a high peak SNR but a Fair or Poor propagation score if activity was sparse or concentrated in the distant past.

Decoder Status

Shows the live status of all decoder pipelines:

  • External FT — jt9/wsprd external processes (requires FT_EXTERNAL_ENABLE=1)
  • CW decoder — Morse code internal decoder
  • SSB / Voice ASR — voice occupancy detector + Whisper ASR pipeline (queue depth shown)
  • PSK decoder — PSK31/63 (not yet implemented)
  • Direwolf KISS (APRS) — APRS packet ingest via Direwolf TCP KISS
  • Last decoder event — timestamp of the most recent event from any source
  • AGC — automatic gain control state

Logs

  • 4ham FrontEnd logs — browser-side diagnostic log.
  • 4ham Server logs — backend log stream (last N lines).

Retention policy

When the total number of stored events exceeds the configured threshold (default 500,000), the system automatically:

  1. Exports all events to a CSV file in the data/exports/ directory.
  2. Purges older events, keeping only the most recent 50,000 (configurable).

This runs at most once every 24 hours. You can trigger it manually via the Admin Config panel.

12 · Academic Analytics dashboard

Purpose

Academic Analytics is an aggregated analysis tool designed to answer concrete questions about amateur radio activity:

  • Which bands are most active right now, today, or over the last 30 days?
  • At what UTC hours does propagation peak on each band? (essential for planning DX operations)
  • What is the real quality of propagation — not just a single SNR peak, but consistency over time?
  • Which stations are most active in the analysed period?
  • How has propagation evolved over hours or days?

Unlike the main dashboard (which shows real-time, event-by-event data), Academic Analytics aggregates and synthesises large volumes of data to reveal patterns and trends not visible in the individual event stream. It is especially useful for identifying the best times and bands for DX, post-session or contest analysis, academic work on HF propagation, and comparing activity across bands and modes over days or weeks.

Access

Accessible via the Data Analysis button in the toolbar (opens in a new tab), or directly at http://<host>:8000/4ham_academic_analytics.html. Data auto-refreshes every 60 seconds. The header shows the last query timestamp and a countdown to the next refresh.

Charts — how to interpret

Each chart has an "i" icon in the header — hover for a description. Hover over chart elements (bars, cells, line points) for a tooltip with exact values.

Event Time Series

Area chart showing event volume over time (by minute, hour or day depending on period resolution).

  • Peaks = moments of high activity — likely a propagation opening or contest.
  • Valleys = periods without activity — propagation closed or scan stopped.
  • Cyclic pattern (peaks at the same hours across days) = normal daily solar cycle — HF propagation rises after sunrise, falls at night.
  • Rising trend over days = improving solar conditions.

Distribution by Band and Mode

Stacked bar chart — one bar per band, divided by mode colours: ■ SSB, ■ FT8, ■ FT4, ■ CW, ■ WSPR.

  • Tall bars = very active bands in the period.
  • Missing bars = band had no activity (propagation closed or not scanned).
  • Bar composition: if dominated by one colour, only that mode was active on the band.
  • Compare heights between bands to decide where to operate.
💡 Only band+mode pairs that had a decoder running appear. If SSB was never scanned on 12m, no SSB events will show for 12m — this is intelligent filtering, not a bug.

Hour of Day × Band — Heatmap Pro

Interactive matrix: 24 rows (UTC hours 0–23) × columns (bands). Colour intensity = event volume at that hour+band intersection.

  • Bright/white cells = heavy activity at that hour+band.
  • Dark cells = little or no activity.
  • Bright horizontal rows = hours with activity across many bands (e.g. 14–18h UTC is typically peak HF propagation).
  • Bright vertical columns = bands active during many hours (consistent propagation).
  • Marginal bars: top (Σ band) = total per band; right (Σ hour) = total per hour.
  • Cross-highlighting: hover on a cell to highlight its full row and column for easy cross-reading.

Practical use: find at which UTC hours your preferred band is most active. Example: if cell [15h UTC, 20m] is bright, that is a good time to operate 20m.

Top Callsigns in Period

Horizontal bar chart of the top 20 callsigns sorted by total hits.

  • High-hit callsigns = stations consistently active with good propagation to your receiver.
  • Useful for spotting beacons, contest stations, or DX super-stations.
  • If your own callsign appears it confirms your signal is getting through.

Propagation Score by Band

Vertical bar chart showing the propagation score (0–100) for each band, based on confirmed decodes only — events with a verified callsign, regardless of mode. A confirmed decode provides a callsign, SNR, and often a grid reference — the reliable basis for assessing a real propagation path. Events without a callsign indicate band occupancy only and do not contribute to this score.

  • ≥ 70 = excellent — consistent decoding, good SNR, many callsigns.
  • 50–69 = good — reliable conditions.
  • 30–49 = fair — marginal signals, intermittent decoding.
  • < 30 = poor — few or no decodes.
  • Compare bars across bands to choose the best band to operate right now.

Propagation Time Trend

Line chart showing how the global propagation score varied over time. Score is derived from confirmed decodes only — events with a verified callsign, regardless of mode. Events without a callsign provide band occupancy data only and are not reflected in this chart. A dashed horizontal line marks the period average.

  • Line above average = conditions better than usual at that moment.
  • Line below average = conditions worse than usual.
  • Rising trend through the day = propagation improving (typical morning → solar peak).
  • Falling trend = propagation deteriorating (typical after sunset on HF).
  • Near-flat, high line = stable and good conditions — ideal for operation.

13 · QTH-Centric Propagation Map

The propagation map is an orthographic 3D globe centred on your QTH, embedded in the Academic Analytics dashboard. It combines two data layers: band-by-band ionospheric zone predictions from NOAA SWPC, and your actual confirmed SDR contacts.

Map elements

ElementDescription
Coloured zones NOAA-predicted propagation coverage per band. Three intensity layers — Strong (opaque), Moderate (semi-transparent), Fringe (faint) — expand toward the subsolar point and shrink on the night side, controlled by D-layer absorption and the ionospheric model.
Day/night terminator Dashed yellow line separating the sunlit and dark hemispheres, computed from the current subsolar point. The dark hemisphere receives an overlay indicating reduced skip propagation.
Dots & arcs Confirmed SDR contacts from the selected time window: a dot at the decoded station's geographic position, and a great-circle arc from your QTH when a grid locator is available.
Graticule Latitude/longitude grid lines with degree labels. Both the grid labels and band distance labels scale with zoom level.

Band buttons

Vertical band buttons on the left of the map toggle individual bands on or off. An active band is shown in its unique colour; an inactive band appears white. Selection is persisted in sessionStorage and restored on the next visit. No bands are selected by default — click any band to reveal its propagation zones.

Map controls

ActionEffect
DragRotate the globe to any orientation.

Legend

Two panels sit below the globe:

  • Left: QTH position, total confirmed contacts in the selected period, anomaly count.
  • Right: Zone intensity swatches (Strong / Moderate / Fringe), Night hemisphere swatch, note that each band uses its own unique colour.

Ionospheric model (zone boundaries)

Zone extent and intensity are computed by the backend from NOAA SWPC data:

17 · APRS decoding — VHF pipeline

4ham includes native APRS decoding from RF, using the RTL-SDR dongle tuned directly to the IARU Region 1 VHF APRS frequency (144.800 MHz). No internet connection is used — all decoding is performed locally from the signals received at the QTH.

Pipeline architecture

When you select APRS mode, the pipeline consists of three chained components:

RTL-SDR → rtl_fm → Direwolf → KISS TCP → 4ham

1. rtl_fm — FM demodulator

  • Role: "The radio" — tunes the RTL-SDR to 144.800 MHz and demodulates the NFM signal into PCM audio.
  • Responsibility: Receives the RF signal from the USB dongle, applies frequency demodulation (NFM), and produces a digital audio stream (22 050 Hz, mono, 16-bit).
  • Why rtl_fm: It is a native C tool from the rtl-sdr project, tested and proven in thousands of APRS iGates worldwide. Unlike a Python demodulator, it guarantees low latency and reliable DSP quality.

2. Direwolf — Software TNC

  • Role: "The decoder" — receives the PCM audio and decodes AX.25/APRS packets.
  • Responsibility: Decodes the AFSK 1200 baud tones (mark=1200 Hz, space=2200 Hz), extracts AX.25 frames (source and destination callsigns, APRS payload), and makes them available via the KISS protocol on a TCP port (default: 8001).
  • Why Direwolf: It is the reference software TNC for APRS on the Linux ecosystem. In KISS mode it acts as a TCP server that 4ham queries for decoded packets.

3. KISS TCP — Transport protocol

  • Role: "The postman" — delivery protocol between Direwolf and 4ham.
  • Responsibility: Transports raw AX.25 frames from Direwolf to the 4ham TCP client, using frames delimited by 0xC0 with escape sequences.
  • What happens in 4ham: The direwolf_kiss.py module connects to Direwolf's KISS server, receives the frames, parses AX.25 addresses (7-byte encoded callsigns), and extracts the APRS payload (position, symbol, comment). Each valid packet is recorded as an event in the database.

Behaviour when switching modes

When you leave APRS mode and switch back to another scan mode (e.g. FT8 on 20 m), 4ham:

  1. Stops the rtl_fm and Direwolf processes
  2. Releases the RTL-SDR dongle
  3. Reopens the SDR via SoapySDR for normal HF/VHF scanning

This cycle is automatic and transparent — simply select the desired scan mode.

18 · FAQ & troubleshooting

The waterfall is blank / shows "No live spectrum data"

This appears only when a scan is running but no FFT frames arrive. Check:

  • The SDR device is connected and recognized (rtl_test or equivalent).
  • No other process (e.g. SDR#, GQRX) is using the device.
  • The backend is running — check Server logs.

FT8/FT4 markers appear and then disappear

This is expected behaviour. Markers expire after 3 decode-window periods of inactivity (FT8: 45 s, FT4: 23 s). If no station is decoded on that band/mode during that time, the marker is removed. A new decode will re-create it immediately.

CW markers disappear too fast

CW markers expire after 45 s (1.5 × the default 30 s dwell). Increase the CW Dwell value to stay longer on each frequency; the marker TTL will remain at the backend's configured value.

No callsigns decoded in FT8/FT4

  • The internal FT decoder is built-in and requires no external binary. Enable it with FT_INTERNAL_ENABLE=1.
  • If using the external decoder (FT_EXTERNAL_ENABLE=1), ensure jt9 (from WSJT-X) is installed and on the system PATH.
  • Weak propagation conditions may result in zero decodes even with a clear spectrum.

The retention toast appears at every startup

Fixed in v0.6.2. The system now persists the last-run timestamp and skips the retention check if it ran within the last 24 hours. If you are still seeing this, upgrade to v0.6.2 or later.

How do I access the API documentation?

The backend exposes an OpenAPI UI at http://<host>:8000/api/docs (SwaggerUI) and /api/redoc (ReDoc).

Voice Signature events appear but no callsign is decoded

Voice Signature means the system detected SSB voice activity but Whisper could not resolve a callsign from the audio. This is normal — SSB callsign extraction depends on audio quality, noise level, operator accent, and whether a callsign was actually spoken. Make sure Whisper ASR is enabled in Admin Config. Even with Whisper active, many transmissions will remain as Voice Signature.

Events card is flooded with occupancy events

Since v0.8.4, the SSB pipeline has built-in flood protection: 30 s debounce per 2 kHz bucket, an 8 dB SNR floor gate, and VOICE DETECTED waterfall markers only on ASR-confirmed voice. If the events card still feels busy:

  • Make sure you are on the default Show All filter — it hides raw occupancy and only shows callsign-type events.
  • If you selected All + Occupancy (raw), raw detections from every scan step will appear. Switch back to Show All.
  • Use SSB Callsign Detected to see only Whisper-resolved callsigns.

Where is the data stored?

All events are stored in data/events.sqlite. Automatic exports go to data/exports/. Manual exports from the Search & Export panel are also saved there.
