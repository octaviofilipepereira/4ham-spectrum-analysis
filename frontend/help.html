1 · Overview

4ham Spectrum Analysis is a real-time spectrum monitoring and digital-mode decoding system for amateur radio, built around an SDR (Software Defined Radio) receiver. It continuously scans configured HF bands, decodes digital signals (FT8, FT4, WSPR, CW) and SSB occupancy, logs every detected event, and displays all activity on a live waterfall and propagation map.

The system has two parts:

  • Backend — Python/FastAPI server that drives the SDR, runs the decoders and stores events in SQLite.
  • Frontend — Single-page web application served by the backend; accessible on the local network at http://<host>:8000.

3 · Scan modes

Only one mode runs at a time on one band. Select the mode before (or during) a scan using the mode buttons.

ModeWindowMarker TTLNotes
FT8 15 s45 s Most popular weak-signal HF mode. Decoded with jt9. Shows callsign markers on the waterfall dial frequency.
FT4 7.5 s23 s Faster variant of FT8, also decoded with jt9. Higher throughput, slightly less sensitivity.
WSPR 120 s360 s Very slow beaconing mode, typically below the noise floor. 2-minute decode windows.
CW dwell 30 s45 s Morse code decoder. Steps across the band in configurable frequency steps. CW Step (Hz) and CW Dwell (s) parameters appear when CW is selected.
SSB continuous20 s (SSB) / 45 s (Voice) SSB voice detection with optional Whisper ASR transcription. Events show Voice Confirmed (voice only), Voice Transcript (has transcript, no callsign — click TXT to see it), or the resolved callsign. Confirmed voice activity produces VOICE DETECTED markers (black-and-gold, 45 s TTL) on the waterfall. Since v0.8.4: 30 s debounce per 2 kHz bucket, 8 dB SNR floor gate, and markers only on ASR-confirmed voice. Enable Whisper in Admin Config.

CW-specific parameters

ParameterDefaultDescription
CW Step (Hz)6500 HzHow many Hz the decoder moves each dwell interval. Smaller steps = more resolution but slower full-band sweep.
CW Dwell (s)30 sTime spent listening at each frequency step before moving on.

SSB-specific parameters

ParameterDefaultDescription
SSB Max Holds0 (auto) Maximum number of times per sweep pass the scanner may pause (hold) on an active SSB frequency. 0 = auto-adaptive (~1 hold per 50 kHz of band span, min 4, max 12). Setting a fixed value limits how long the scanner can be “stuck” on busy frequencies before completing the full band pass. Applied at scan start only — has no effect while a scan is already running.
💡 FT8/FT4 dial frequency — the decoder locks onto the standard dial frequency for each band (e.g. 14.074 MHz for FT8 20m). A single waterfall marker represents the last decoded callsign for that band/mode combination.

4 · Waterfall & spectrum display

Layout

  • Spectrum graph (top strip) — instantaneous power spectrum, updated at ~5–10 fps.
  • Frequency ruler — shows frequency labels aligned to the waterfall below.
  • Waterfall canvas — scrolling time-frequency display. Newer data is at the top; time flows downward.
  • Mode overlay — coloured labels pinned to known mode frequencies (e.g. FT8 dial, WSPR, CW segment). The label contains mode, last decoded callsign, and SNR.

Interacting with the waterfall

  • Mouse / touch drag — pan horizontally when zoomed in.
  • Zoom slider — horizontal zoom from ×1 (full band view) to ×16.
  • Mouse hover — VFO bar updates to show the frequency under the cursor, plus live SNR.
  • Go to field — type a frequency in MHz and press Enter or click GO to centre the view on that frequency.

Marker colours

ColourMeaning
■ BlueFT8 decoded callsign marker
■ GreenFT4 decoded callsign marker
■ OrangeWSPR decoded marker
■ Pink/purpleCW detected signal marker
■ Light blueSSB occupancy marker

Transition overlay

When switching bands or modes, a transition overlay appears over the waterfall while the backend reconfigures. This is normal — the spinner disappears once the new spectrum stream arrives.

6 · SSB Voice Signature & Whisper ASR

Since v0.8.0, the system can detect and transcribe SSB voice transmissions using VAD (Voice Activity Detection) and OpenAI Whisper automatic speech recognition.

How it works

  1. During scanning, SSB occupancy is detected by bandwidth and power heuristics.
  2. If voice activity is confirmed (VAD), a 15-second audio segment is captured.
  3. If Whisper ASR is enabled, the audio is transcribed in a background thread.
  4. The transcript is parsed for callsign patterns (NATO phonetics, direct callsigns, "CQ" calls).
  5. If a callsign is resolved it appears with the callsign; if a transcript exists but no callsign, it appears as Voice Transcript (click TXT to see the text); otherwise it appears as Voice Confirmed.

SSB event labels

SSB events are shown with one of three labels depending on the detection result:

LabelMeaning
Voice ConfirmedVoice activity detected by VAD but no Whisper transcript available.
Voice TranscriptWhisper produced a transcript (shown via the TXT button) but no valid callsign was resolved.
CallsignWhisper transcript contained a valid callsign — shown directly in the callsign column.

All three types are included in the "Show All" filter by default.

VOICE DETECTED waterfall markers

Confirmed SSB voice activity (ASR-verified) produces real-time VOICE DETECTED markers on the waterfall overlay. These markers have a distinctive black background with gold text and a 45-second TTL — long enough to survive natural pauses between SSB "overs" without flickering. Each new voice detection resets the timer. Since v0.8.4, markers are only created when Whisper ASR confirms actual voice; occupancy-only detections no longer produce waterfall markers.

Enabling Whisper ASR

Open Admin Config and scroll to SSB Voice Transcription (ASR). Tick the checkbox and click Save ASR setting. Whisper requires the openai-whisper Python package (~700 MB download). You can install it via the interactive installer or manually: pip install openai-whisper.

💡 On Raspberry Pi or low-end hardware, each 15 s audio segment takes 2–5 s to transcribe. Transcription runs in a background thread and does not block scanning.

7 · Metrics explained

SNR (Signal-to-Noise Ratio)

Instantaneous measurement of a single signal relative to the noise floor, in dB.

SNR rangeInterpretation
< 8 dBSignal rejected — too weak to decode reliably.
8–15 dBWeak but decodable.
15–25 dBStrong signal.
> 25 dBExcellent signal strength.

The SNR shown next to each band in the propagation map is the maximum peak SNR recorded in the last 60 minutes — not a current reading.

Propagation Score (0–100)

An aggregated quality score for a band, computed from all events in the last 60 minutes. Each event contributes a weighted score based on SNR, decode confidence, and how recently it occurred. Recent, high-confidence events with strong SNR push the score up.

ScoreClassification
≥ 70🟢 Excellent
50–69🟡 Good
30–49🟠 Fair
< 30🔴 Poor
⚠ A band can show a high peak SNR but a Fair or Poor propagation score if activity was sparse or concentrated in the distant past.

8 · Export & reports

Search & Export Data

Accessed via Logs & Reports → Search & Export Data. Apply filters and then export matching events as:

  • CSV — comma-separated, compatible with spreadsheets.
  • JSON — structured data for programmatic use.
  • PNG — screenshot of the current events table view.

Session Summary

Opens a modal with per-band statistics for the current session: total events, mode breakdown, peak SNR, and propagation score history.

Decoder Status

Shows the live status of all decoder pipelines:

  • External FT — jt9/wsprd external processes (requires FT_EXTERNAL_ENABLE=1)
  • CW decoder — Morse code internal decoder
  • SSB / Voice ASR — voice occupancy detector + Whisper ASR pipeline (queue depth shown)
  • PSK decoder — PSK31/63 (not yet implemented)
  • Direwolf KISS (APRS) — APRS packet ingest via Direwolf TCP KISS
  • Last decoder event — timestamp of the most recent event from any source
  • AGC — automatic gain control state

Logs

  • 4ham FrontEnd logs — browser-side diagnostic log.
  • 4ham Server logs — backend log stream (last N lines).

Admin Config — Buttons

ButtonWhat it does
Refresh Devices Forces a new SoapySDR device enumeration, bypassing the 300-second cache. Populates the device dropdown and automatically applies the correct gain and sample rate defaults for the detected device (RTL-SDR: gain 30, 2.048 MS/s; HackRF: gain 20, 2 MS/s; AirSpy: gain 20, 2.5 MS/s).
Save device Persists the current Device Configuration fields (device class, PPM, frequency offset, gain profile) to the database. Changes made in the form are only local until this button is clicked.
Save audio Persists the current Audio Configuration fields (input/output device, sample rate, RX/TX gain) to the database.
Test Config Validates the current configuration without saving. Checks: SoapySDR device availability, audio tools present on the system (arecord, aplay, pactl, pw-cli), and that sample rate / gain values are within accepted ranges. Reports pass or fail with specific details in a toast notification.
Auto-detect Device Two-step automated setup wizard. Step 1 (dry run): queries the backend for required packages, shows a confirmation dialog with current status and missing dependencies. Step 2 (if approved): installs missing system packages via sudo / pkexec, detects the SDR device, applies the recommended profile (gain, sample rate, PPM, gain profile) to the form fields, and saves the configuration to the database.
Auto-detect Audio Queries the backend for available audio devices (PipeWire / PulseAudio / ALSA). Fills in the Audio Configuration fields with the detected input/output device names and sample rate. Does not save automatically — click Save audio afterwards to persist.
Purge invalid events Asks for confirmation, then deletes from the database all occupancy and callsign events that are incomplete or malformed (missing timestamp, invalid frequency, null/unknown callsign, etc.). Updates event counters in the UI after completion. Reports the number of events deleted.
Reset defaults Asks for confirmation, then restores the application's default settings (active modes, summary options, and other general settings). Does not affect stored events, custom bands, device configuration, or audio configuration.
Reset total ⚠️ Destructive. Asks for confirmation, then deletes all settings and custom bands from the database (DELETE FROM settings, DELETE FROM bands), clears browser local storage, and reloads the page. Equivalent to a clean installation state. Stored events are not affected.

Retention policy

When the total number of stored events exceeds the configured threshold (default 500,000), the system automatically:

  1. Exports all events to a CSV file in the data/exports/ directory.
  2. Purges older events, keeping only the most recent 50,000 (configurable).

This runs at most once every 24 hours. You can trigger it manually via the Admin Config panel.

11 · Academic Analytics dashboard

Accessible via the Data Analysis button in the toolbar (opens in a new tab), or directly at http://<host>:8000/4ham_academic_analytics.html.

The dashboard provides aggregated views of collected event data for academic and analytical purposes:

  • Activity Timeline — event count over time, bucketed by hour or day.
  • Band Distribution — pie or bar chart showing events per amateur band.
  • Hour of Day × Band (Heatmap Pro) — interactive heatmap with cross-highlight on hover, marginal totals per row/column, and colour-scaled cells for quick pattern recognition.
  • Propagation Map — geographical distribution of decoded callsigns.

Data is fetched from /api/analytics/academic and auto-refreshes every 60 seconds. A period selector (1 h / 6 h / 24 h / 7 d / 30 d / All) filters the time window. Each chart has a ? helper icon that shows a description on hover.

The .env file in the project root controls service environment variables (host, port, decoder flags). It is created automatically on first install and is not committed to version control.

12 · Desktop launcher

On systems with a graphical desktop (GNOME, Cinnamon, XFCE, KDE, MATE, LXQt), the installer offers to create a desktop shortcut. Double-clicking it opens an interactive terminal menu:

  • 1) Open Dashboard in Browser — opens http://127.0.0.1:8000/ (only if the server is running).
  • 2) Start Server — starts the backend process.
  • 3) Restart Server — stops and restarts the backend.
  • 4) Stop Server — stops the backend process.
  • 0) Exit — closes the launcher.

The launcher script is scripts/4ham_launcher.sh and can also be run directly from a terminal. The shortcut is placed on the Desktop and in ~/.local/share/applications/. The uninstaller removes it automatically.

13 · FAQ & troubleshooting

The waterfall is blank / shows "No live spectrum data"

This appears only when a scan is running but no FFT frames arrive. Check:

  • The SDR device is connected and recognized (rtl_test or equivalent).
  • No other process (e.g. SDR#, GQRX) is using the device.
  • The backend is running — check Server logs.

FT8/FT4 markers appear and then disappear

This is expected behaviour. Markers expire after 3 decode-window periods of inactivity (FT8: 45 s, FT4: 23 s). If no station is decoded on that band/mode during that time, the marker is removed. A new decode will re-create it immediately.

CW markers disappear too fast

CW markers expire after 45 s (1.5 × the default 30 s dwell). Increase the CW Dwell value to stay longer on each frequency; the marker TTL will remain at the backend's configured value.

No callsigns decoded in FT8/FT4

  • The internal FT decoder is built-in and requires no external binary. Enable it with FT_INTERNAL_ENABLE=1.
  • If using the external decoder (FT_EXTERNAL_ENABLE=1), ensure jt9 (from WSJT-X) is installed and on the system PATH.
  • Weak propagation conditions may result in zero decodes even with a clear spectrum.

The retention toast appears at every startup

Fixed in v0.6.2. The system now persists the last-run timestamp and skips the retention check if it ran within the last 24 hours. If you are still seeing this, upgrade to v0.6.2 or later.

How do I access the API documentation?

The backend exposes an OpenAPI UI at http://<host>:8000/api/docs (SwaggerUI) and /api/redoc (ReDoc).

Voice Signature events appear but no callsign is decoded

Voice Signature means the system detected SSB voice activity but Whisper could not resolve a callsign from the audio. This is normal — SSB callsign extraction depends on audio quality, noise level, operator accent, and whether a callsign was actually spoken. Make sure Whisper ASR is enabled in Admin Config. Even with Whisper active, many transmissions will remain as Voice Signature.

Events card is flooded with occupancy events

Since v0.8.4, the SSB pipeline has built-in flood protection: 30 s debounce per 2 kHz bucket, an 8 dB SNR floor gate, and VOICE DETECTED waterfall markers only on ASR-confirmed voice. If the events card still feels busy:

  • Make sure you are on the default Show All filter — it hides raw occupancy and only shows callsign-type events.
  • If you selected All + Occupancy (raw), raw detections from every scan step will appear. Switch back to Show All.
  • Use SSB Callsign Detected to see only Whisper-resolved callsigns.

Where is the data stored?

All events are stored in data/events.sqlite. Automatic exports go to data/exports/. Manual exports from the Search & Export panel are also saved there.
